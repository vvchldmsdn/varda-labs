# Auth/Tenant Phase 0 Preflight

Last updated: 2026-07-10

Status: completed read-only preflight. This document and the companion audit do
not authorize an auth provider, schema migration, owner backfill, RLS policy,
query behavior change, user write path, or recommendation implementation.

The provider/session decision that follows this preflight is recorded in
`docs/auth-identity-session-strategy.md`.

## Outcome

The current deployment is still a single imported portfolio behind Basic Auth.
Basic Auth is an access gate, not an identity provider, and account codes such
as `brokerage`, `isa`, `irp`, and `all` are portfolio partitions, not tenants.

The canonical future contract is:

| Concern | Decision |
| --- | --- |
| Internal user table | `app_users` |
| Internal user key | `app_users.id uuid` |
| Provider mapping table | `auth_identities` |
| User-owned row key | `owner_user_id uuid NOT NULL REFERENCES app_users(id)` |
| Existing varchar owners | Legacy evidence only; not internal UUIDs |
| Initial imported data | Assign to one explicitly approved initial app user |
| Shared market data | One shared copy; no tenant owner column |
| Basic Auth | Deployment access only; never supplies `owner_user_id` |

No current row is canonical-owner ready. The read-only database audit reports
757 user-owned rows, all of which require an approved owner migration before
multi-user access can be enabled.

## Reproducible Audit

Run:

```powershell
npm run audit:tenant-ownership
```

The audit performs five `SELECT` statements, classifies every public base
table, reports only sanitized counts/metadata, and fails if a table is missing
from the policy. It never prints owner values and never writes to the database.

Observed on 2026-07-10:

- 22 public base tables
- 0 foreign keys
- 14 `user_owned` tables, 757 rows
- 7 `shared_reference` tables, 27,806 rows
- 1 `admin_system` table, 65 rows
- 0 `unresolved` tables
- 0 rows with the final non-null UUID owner contract

Counts are operational evidence, not migration guards by themselves. A future
migration must re-run and explicitly approve fresh counts.

## Canonical Identity Model

Phase 1B fixed the `app_users` fields:

| Field | Contract |
| --- | --- |
| `id` | UUID primary key generated by varda-labs |
| `status` | `provisioning`, `active`, or `disabled` lifecycle state |
| `role` | `user` or `admin`; authorization metadata only |
| `created_at`, `updated_at` | Internal timestamps |

No email or provider profile is copied into `app_users`.

Phase 1B fixed the `auth_identities` fields:

| Field | Contract |
| --- | --- |
| `id` | UUID primary key |
| `app_user_id` | Non-null FK to `app_users.id` |
| `provider` | Provider namespace |
| `provider_subject` | Provider-stable external subject |
| `status`, `disabled_at` | Active/disabled mapping lifecycle |
| `created_at`, `updated_at` | Internal timestamps |
| unique key | `(provider, provider_subject)` |

The product must resolve a provider subject to an internal UUID on the server.
External provider IDs, email addresses, Basic Auth usernames, Base44 owner
strings, and `base44-import` labels must never become row ownership keys.

## Initial Owner Backfill Plan

The future migration must be dry-run/approval gated and use an explicit
`--initial-owner-id <uuid>` (or equivalent reviewed configuration). It must not
invent or infer the owner.

Recommended sequence:

1. Re-run row/null/distinct-owner guards and stop on unexpected values.
2. Expand with `app_users`, `auth_identities`, and nullable
   `canonical_owner_user_id uuid` on all 14 user-owned tables.
3. Preserve every existing varchar `owner_user_id` and `assets.created_by_id`;
   do not cast, overwrite, or infer identity from them.
4. Create one separately reviewed `app_users` row for the imported portfolio.
5. Freeze user-owned writers or deploy verified canonical-owner dual-write.
6. Backfill only null canonical columns with the explicit initial UUID in one
   guarded transaction.
7. Validate row counts, null counts, relationship ownership, and uniqueness.
8. Add owner-aware unique indexes and `NOT VALID` FKs, then validate them.
9. Deploy and prove owner-filtered application reads/writes.
10. Only in the final contract migration, set non-null and rename old varchar
    owners to `legacy_owner_user_id` and canonical owners to `owner_user_id`.

Per-table strategy:

| Table group | Backfill action |
| --- | --- |
| `accounts`, `asset_groups`, `asset_group_members`, `goals`, `transactions`, `fixed_transactions`, `monthly_incomes` | Keep varchar owner unchanged; add/backfill the parallel canonical UUID; rename the varchar only at final contract |
| `assets` | Add/backfill the parallel canonical UUID; retain `created_by_id` as deprecated legacy evidence |
| `event_ledger_entries`, `market_regime_daily`, snapshots, `settings` | Add/backfill the parallel canonical UUID, then validate counts and same-owner relations |
| Shared reference tables | No owner backfill and no tenant duplication |
| `market_data_sync_runs` | No user owner; keep behind an admin/system role |

The exact Phase 1B transition matrix and staged drafts supersede the earlier
direct-cast concept and are recorded in
`docs/auth-tenant-phase1b-migration-plan.md`.

## Recommendation Ownership Contract

Future `recommendation_runs` and `recommendation_items` are user-owned:

- both require `owner_user_id uuid NOT NULL`;
- item owner must equal its parent run owner;
- `account` and nullable `account_id` are portfolio scope, not tenant identity;
- recommendation reads start with owner and then apply account/date filters;
- import, if later approved, requires the explicit initial owner;
- execution remains a separate owner-checked command/event path.

## Two-User Isolation Fixture Plan

Create deterministic users A and B with deliberately colliding account codes,
ticker symbols, dates, and labels. This catches missing owner predicates that
would be hidden by globally unique fixture values.

Required coverage:

1. Each user owns accounts, assets/groups, settings, events, snapshots, and
   regime rows; shared market rows exist only once.
2. Dashboard, today movement, history, structure, risk, and market-context
   query helpers return only the selected user's financial rows.
3. Shared prices, FX, ETF, benchmark, and factor evidence is available to both
   users without being duplicated.
4. Counts, labels, totals, and unmatched snapshot fallbacks contain no
   cross-user values.
5. User writes reject a missing owner and ignore/reject a browser-supplied
   owner; ownership comes from server context.
6. Recommendation runs/items require an owner and reject parent/item owner
   mismatch.
7. An explicit admin fixture may see cross-user operational aggregates.
8. A later RLS integration suite sets users A and B in separate transactions
   and proves the same isolation at the database layer.

## RLS ADR Direction

RLS remains deferred until app-level owner filtering works and the Neon/Drizzle
transaction behavior is proven.

Future role model:

- migration/owner role: owns schema and runs reviewed migrations;
- application role: does not own tables and has no `BYPASSRLS`;
- admin job role: separate from user sessions and explicitly authorized;
- public browser: never connects to Postgres directly.

Future request identity direction:

1. Server session resolves provider identity to `app_users.id`.
2. App-level queries always include `owner_user_id` predicates.
3. In one database transaction, set a transaction-local identity with
   `set_config('app.current_user_id', <uuid>, true)`.
4. RLS `USING` and `WITH CHECK` compare `owner_user_id` with a null-safe cast of
   that setting.
5. Prove Neon HTTP/Drizzle keeps the setting and protected query in the same
   transaction/connection before any policy is enabled.
6. Shared tables receive read grants without owner RLS; admin/system tables are
   not granted to the application role.

The future policy shape may use:

```sql
owner_user_id = nullif(current_setting('app.current_user_id', true), '')::uuid
```

This is an ADR direction only, not executable migration SQL.
