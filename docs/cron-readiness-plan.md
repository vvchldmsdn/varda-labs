# Cron Readiness Plan

Last updated: 2026-07-08

This is a docs-only plan. It does not enable Vercel Cron, add `vercel.json`,
change route handlers, change database schema, or broaden any write path.

## Current Status

The production manual market cycle has now succeeded on two consecutive cycles:

- 2026-07-07 KST cycle
- 2026-07-08 KST cycle

Both cycles followed the same guarded runbook:

1. Production daily snapshot dry-run.
2. Inspect `closeSyncPlan` and required fresh-close coverage.
3. Run KIS close dry-runs by reviewed batch.
4. Run guarded KIS close writes with `dryRun=false&confirmWrite=true`.
5. Run duplicate audit for `asset_price_snapshots`.
6. Run market-sync metadata secret audit.
7. Run daily snapshot dry-run again.
8. Write the daily snapshot only after fresh-close coverage is complete.
9. Run post-write daily snapshot dry-run and confirm update-only behavior.

The manual path is now validated enough to design automation. It is not enough
to enable Cron directly.

## Vercel Cron Constraints

Current platform constraints to design around:

- Vercel Cron triggers an HTTP `GET` request to the production deployment URL
  using the `path` configured in `vercel.json`.
- Cron-triggered requests include `vercel-cron/1.0` as the user agent and an
  `x-vercel-cron-schedule` header.
- Cron expressions are evaluated in UTC.
- `CRON_SECRET`, when configured, is sent by Vercel as an Authorization bearer
  token.
- Cron jobs have the same duration limits as Vercel Functions.
- Failed Cron invocations are not retried automatically.
- Duplicate, missed, or overlapping invocations must be handled by application
  logic with idempotency and locking.
- Production Cron jobs do not follow redirects.
- Local framework dev servers do not exercise production Cron behavior.
- Hobby-plan Cron timing can run once per day at any point within the specified
  hour; non-Hobby timing should still be treated as approximate enough to
  require idempotent handlers.

References:

- https://vercel.com/docs/cron-jobs
- https://vercel.com/docs/cron-jobs/manage-cron-jobs

## Current Blockers

Cron should stay disabled until these are resolved:

- Existing admin write routes are `POST`-only:
  - `POST /api/admin/market/prices/sync`
  - `POST /api/admin/snapshots/daily`
- Vercel Cron invokes `GET` paths, so the existing admin routes cannot be wired
  directly to Cron.
- There is no `vercel.json`.
- KIS actual close writes are intentionally manual and capped at `limit <= 5`.
- The KIS close route has a 90 second cooldown, and that cooldown also affects
  dry-runs.
- A single Cron invocation must not sleep through repeated 90 second cooldowns
  just to process multiple KIS batches.
- Overlapping run prevention is not implemented as a dedicated Cron contract.
- A durable run status model for partial close sync progress is not designed.

## Future Design Decisions

Before implementation, decide:

- Whether to add a separate cron-only `GET /api/cron/...` controller surface.
- Whether Phase 1 Cron routes are dry-run/preflight only.
- How the controller derives required close targets from daily snapshot
  `closeSyncPlan`.
- How to process 15+ close targets without abusing the manual `limit <= 5`
  route policy.
- Whether KIS close automation uses repeated scheduled invocations, a durable
  job table, or another queue/controller design.
- How partial KIS failures are recorded and exposed without allowing partial
  daily snapshot writes.
- How to prevent overlapping Cron runs for the same market cycle.
- Whether `market_data_sync_runs` is sufficient for run status, or whether a
  future job table is required before automation.
- How the snapshot route verifies complete close coverage before any actual
  snapshot write.

## Non-Negotiable Safety Rules

- No partial daily snapshot write.
- No `assets.current_price` update from the close-sync Cron path.
- No KIS token, app key, app secret, authorization header, request header, or raw
  provider response stored in Postgres.
- No ready-to-run `confirmWrite=true` URL generated by any planner.
- No cleanup, delete, destructive backfill, or repair path in Cron.
- No Cron route should expose secrets or raw provider error payloads.
- Cron actual writes require separate approval after this design is reviewed.

## Candidate Phases

### Phase 0: Docs-only readiness plan

Capture the constraints, blockers, and safety rules. No code changes.

Status: this document.

### Phase 1: Cron dry-run/preflight route design

Design a `GET` route that reports what the next market cycle would need without
performing writes. It should reuse the same calculation surfaces as manual
dry-runs and preserve the current write guards.

This phase should not add `vercel.json`.

### Phase 2: Cron-safe close sync controller

Design and manually test a controller that can process close sync work without
sleeping through cooldown windows in one serverless invocation. It must record
state, respect KIS limits, and remain idempotent.

### Phase 3: Snapshot dry-run automation

Automate the daily snapshot dry-run only after close sync coverage can be
observed reliably. Actual snapshot writes remain manual.

### Phase 4: Guarded snapshot write automation

Allow snapshot actual writes only when:

- the current cycle is resolved,
- fresh-close coverage is complete,
- duplicate and unmanaged-row checks pass,
- imported Base44 rows are not overwritten,
- the operation is idempotent for already-generated varda rows.

### Phase 5: Vercel Cron enablement

Add `vercel.json` and production Cron schedules only after the route design,
manual controller tests, and safety review are complete. This phase requires a
separate approval.

## Current Recommendation

Do not enable Cron yet. The next implementation step should be a reviewed
cron-only `GET` preflight design, but only after this document is accepted.
Until then, keep using the manual runbook in
`docs/price-sync-and-snapshot-pipeline.md`.
